# Клиентская часть — спецификация доработок

Финальные решения по 8 пунктам доработок клиентской части бота,
собранные на основе ответов владельца. **Пункт 1 (баг статуса подкопа)
выведен в отдельный трек** — по нему нужна предварительная диагностика
реального поведения подкопа на роутере.

Приоритет реализации (от простого к сложному): **5,6 → 3,4 → 7 → 8 → 2**.

---

## Пункт 5. «🔄 podkop restart» → «🔄 Перезагрузить службу»

Простое переименование кнопки и текста подтверждения. Внутренне
действие остаётся тем же (рестарт podkop-сервиса). Во всех местах,
где клиент видит слово «podkop», заменить на «служба» или убрать.

- Кнопка в меню роутера: `🔄 Перезагрузить службу`.
- Подтверждение: `Перезагрузить службу на id<NN>? Интернет моргнёт 5–10 сек.`.
- Успех: `✅ Служба перезапущена`.
- Фейл: `❌ Не удалось перезапустить службу. Лог: <tail>`.

Admin-интерфейс **не меняется** — там оставляем «podkop restart» как
было, технический термин для технаря.

---

## Пункт 6. «♻ Restart router» → «♻ Перезагрузить Роутер (рекомендуется)»

Переименование с пометкой «(рекомендуется)» — статичный текст всегда,
без умной логики по статусу. Умную рекомендацию прикрутим позже
отдельной задачей, когда будет готов детектор (п.1).

Двойное подтверждение сохраняется как сейчас.

## Пункт 3. Отображение IP и страны на info-экране

### Как определяем

**Финальная схема после диагностики на id02 (апрель 2026).**
Используем два эндпойнта, которые стабильно работают на OpenWrt с
busybox `wget` + sing-box/podkop:

- `wget -qT5 -O- https://yandex.ru/internet/api/v0/ip/` — идёт
  **напрямую через провайдера** (yandex.ru в списке `russia_inside`).
  Возвращает строку `"176.15.166.202"` — **IP от провайдера** (включая
  случай CGNAT от 4G-оператора — внутренний адрес роутера не отражает
  настоящий внешний IP, а yandex его видит).
- `wget -qT5 -O- https://ifconfig.co/json` — идёт **через VLESS**
  (ifconfig.co не в `russia_inside`). Возвращает JSON с `ip`,
  `country`, `country_iso`, `asn_org` → **egress IP + страна**.

`api.2ip.ru` не использовать — стабильно возвращает `Operation not
permitted` из-за какой-то фильтрации в sing-box/nftables.

### Определение страны провайдерского IP

`yandex.ru/internet/api/v0/ip/` отдаёт только IP без страны. Страну
бот определяет сам через локальную базу **MaxMind GeoLite2-Country**
(бесплатная, ~3 МБ, качается при `install.sh`, обновляется через
`scripts/update.sh` раз в месяц). Библиотека: `maxminddb` или
`geoip2` для Python.

### Логика определения что подкоп живой

- если `local_ip != egress_ip` → ✅ подкоп маршрутизирует как надо.
- если `local_ip == egress_ip` или один из запросов упал → ⚠ подкоп
  запущен, но трафик идёт мимо VLESS. Показать предупреждение.

Основной детектор статуса — из пункта 1 (см. ниже).

### Кеш

Результаты кешируются в `bot.db` на **10 минут** на роутер. При нажатии
кнопки `ℹ info` чаще — отдаём из кеша. Принудительный рефреш — через
кнопку `🔄 Обновить данные`.

### Формат отображения

```
🏠 От провайдера: 🇷🇺 92.xx.xx.xx (Россия)
🌍 Через VLESS: 🇳🇱 185.xx.xx.xx (Netherlands)
```

Если второй запрос таймаутнулся — показываем `🌍 Через VLESS: ⚠ недоступно`.

## Пункт 4. Краткий VLESS для клиента

Клиент видит **только страну + доменное имя** VLESS-сервера:

```
🔗 VLESS: 🇳🇱 de.example.com
```

**Скопировать полный URL клиент не может.** Полный `vless://...` остаётся
только у админа (через админ-меню `⚙ Настройки → VLESS-пул → просмотр`).

Страна отображается из VLESS-конфига (что прописано в пуле админом).
Реальная страна egress показывается отдельно на info-экране (п.3) —
они могут не совпадать и это нормально.

## Пункт 7. WiFi-меню

### Кнопка

В меню роутера добавляется кнопка `📶 WiFi`.

### Что показывается

```
📶 WiFi настройки id<NN>
━━━━━━━━━━━━━━━━━━━━━
Сеть: MyHomeNet
Пароль: ••••••••••••
Шифрование: WPA2-PSK

[👁 Показать пароль]
[✏️ Изменить]
```

### Логика

- **Радио объединены.** Бот применяет один SSID и один пароль на все
  `wifi-iface` (2.4 GHz + 5 GHz). Бесшовный роуминг.
- **Шифрование всегда WPA2-PSK** (`option encryption 'psk2'`). Клиент
  это не выбирает.
- **Валидация:**
  - SSID: 1–32 байта, любые символы.
  - Пароль: 8–63 символа, любые печатные ASCII.
- **Двойное подтверждение** перед применением:

  ```
  ⚠ ВСЕ подключённые к WiFi устройства отвалятся и должны будут
  переподключиться с новым паролем. Если ты сейчас подключён
  по WiFi — твой телефон тоже отвалится на несколько секунд.
  
  Новая сеть: MyNewNet
  Новый пароль: secret12345
  
  Применить? [✅ Да, применить]  [❌ Отмена]
  ```

- **Guest-сеть** (если есть `wifi-iface[2]`) — **не трогаем в MVP**.
  Редактируем только основную (iface 0 и 1).

### Применение

```bash
# на роутере, через SSH
uci set wireless.@wifi-iface[0].ssid='MyNewNet'
uci set wireless.@wifi-iface[0].key='secret12345'
uci set wireless.@wifi-iface[0].encryption='psk2'
uci set wireless.@wifi-iface[1].ssid='MyNewNet'
uci set wireless.@wifi-iface[1].key='secret12345'
uci set wireless.@wifi-iface[1].encryption='psk2'
uci commit wireless
wifi reload
```

Старое значение SSID/password сохраняем в `bot.db` (как последний известный
snapshot), на случай если вдруг надо показать что было до.

### Лимит

Клиенту: **5 смен WiFi-настроек в сутки**. Админу — без лимита.

### Audit

Пишется в `audit`: `(client_id, router_id, 'wifi_change', {ssid_before, ssid_after}, ts)`.
Пароли в audit **не пишем**.

## Пункт 8. Кастомные правила маршрутизации

### Кнопка

В меню роутера: `📋 Маршрутизация`. Видна и клиенту, и админу.

### Что показывается

```
📋 Домены через VLESS на id<NN>
━━━━━━━━━━━━━━━━━━━━━━━━━━━
Всего: 12 доменов

[➕ Добавить]  [✏️ Изменить]

1. youtube.com         [❌]
2. chatgpt.com         [❌]
3. 2ip.io              [❌]
...
```

### Кто может править

Клиент **и** админ. Клиенту — с лимитами, админу — без.

### Лимиты для клиента

- Максимум **200 доменов** в списке на роутер.
- Не больше **20 добавлений** в сутки.
- Не больше **20 удалений** в сутки.
- Превышение → `❌ Лимит N/день исчерпан, вернись завтра`.

### Валидация домена

Строгая регулярка:

```
^[a-z0-9]([a-z0-9-]{0,61}[a-z0-9])?(\.[a-z0-9]([a-z0-9-]{0,61}[a-z0-9])?)+$
```

**Отклоняем:**
- `https://example.com` → есть протокол
- `www.example.com` → префикс www.
- `example.com:8080` → порт
- `example.com/path` → путь
- `192.168.1.1` → IP-адрес
- `example` → нет точки (не домен)
- `ПРИМЕР.рф` → кириллица/не-ASCII (IDN — отдельная история, пока не поддерживаем)
- пробелы, таб, любые спец. символы

**Принимаем:**
- `example.com`
- `sub.example.com`
- `2ip.ru` (цифры в домене разрешены)
- `m3u8.com`

Текст отказа клиенту:

```
❌ Неверный формат.
Нужно: example.com или sub.example.com
Нельзя: http://, https://, www., порты, пути, IP-адреса, кириллица.
```

### Поток редактирования — меню-буфер

1. Клиент заходит в `📋 Маршрутизация`.
2. В фоне загружается текущий список из роутера (`uci show podkop | grep user_domains`).
3. Клиент добавляет/удаляет в **буфере** (ничего ещё не применено).
4. Все изменения копятся в памяти сессии + отдельная таблица `pending_changes`
   в `bot.db` (чтобы выжить при рестарте бота).
5. При выходе из меню (`⬅ Назад` или `🏠 Меню`) бот показывает diff:

   ```
   📋 Применить изменения на id<NN>?
   
   ➕ Добавлено:
     youtube.com
     chatgpt.com
   
   ➖ Удалено:
     oldsite.com
   
   ⚠ После применения служба перезапустится,
   интернет моргнёт 5–10 сек.
   
   [✅ Применить]  [❌ Отменить изменения]
   ```

6. По `✅ Применить` — бот выполняет на роутере:

   ```bash
   # удаляем
   uci del_list podkop.main.user_domains='oldsite.com'
   # добавляем
   uci add_list podkop.main.user_domains='youtube.com'
   uci add_list podkop.main.user_domains='chatgpt.com'
   # переключаем тип в зависимости от итогового размера списка
   uci set podkop.main.user_domain_list_type='dynamic'   # или 'disabled' если 0
   uci commit podkop
   /etc/init.d/podkop restart
   ```

7. По `❌ Отменить` — буфер чистится, ничего на роутере не меняется.

### Переключатель `user_domain_list_type`

Бот автоматически следит за этим:

- итоговый размер списка 0 → `option user_domain_list_type 'disabled'`
- итоговый размер ≥1 → `option user_domain_list_type 'dynamic'`

Клиент об этой переменной не знает и не видит её.

### Audit

На каждое применение пишется:
`(client_id, router_id, 'routing_change', {added: [...], removed: [...]}, ts)`.

## Пункт 2. «Отправить лог администратору»

### Где кнопка

После выполнения `✅ test (global check)` в ответном сообщении появляется
кнопка `📤 Отправить лог админу`.

### Что отправляется

Полный вывод `test (global check)` + последние 100 строк
`logread | grep -E 'podkop|xray'` — приклеенным текстовым файлом
`log_id<NN>_<timestamp>.txt` (если короткий — прямо в сообщение).

### Как админ получает

**Гибридно:**

1. **Пуш в админ-чат** с превью:
   ```
   🆘 Лог от клиента
   Роутер: id12
   Клиент: Иван (@ivan_tg, id 12345)
   Дата: 2026-04-24 14:35
   
   Превью (первые 5 строк):
     [12:30:01] podkop=STOP
     [12:30:02] xray exited
     ...
   
   [📄 Полный лог]  [✉ Ответить клиенту]
   ```

2. **Копия в меню админа**: `⚙ Настройки → 📥 Логи от клиентов`.
   Там висит список с бейджем `(N новых)` у непрочитанных, можно
   отсортировать по роутеру/клиенту/дате.

### Retention

Логи хранятся **30 дней** в `bot.db` + файлы в
`/var/lib/r-bot/client_logs/id<NN>/`, потом автоматически удаляются
(джоба раз в сутки).

### Лимит

Клиент может отправить **5 логов в сутки**. Превышение →
`❌ Лимит 5/день, вернись завтра или попроси админа связаться с тобой`.

### Audit

`(client_id, router_id, 'send_log_to_admin', {log_id}, ts)`.

После любого write-действия (wifi/routing/reboot) кеш info-экрана

## Пункт 1. Детектор статуса подкопа — РЕШЕНО

### Что было не так

Бот опрашивал `/etc/init.d/podkop status` — команда **всегда** возвращает
`not running` (exit=5), потому что подкоп не procd-демон, а набор
скриптов + nftables-правила + воркер sing-box. `pgrep podkop` тоже
почти всегда пустой по той же причине. `pgrep xray` пустой потому что
рабочий процесс называется `sing-box`, не xray.

### Что использовать в боте

Один SSH-вызов на роутер собирает все нужные сигналы:

```sh
SING=$(pgrep -f "^/usr/bin/sing-box" >/dev/null 2>&1 && echo 1 || echo 0)
TABLE=$(nft list tables 2>/dev/null | grep -q "PodkopTable" && echo 1 || echo 0)
LOCAL_IP=$(wget -qT5 -O- 'https://yandex.ru/internet/api/v0/ip/' 2>/dev/null | tr -d '"')
EGRESS_JSON=$(wget -qT5 -O- 'https://ifconfig.co/json' 2>/dev/null)
echo "SING=$SING"
echo "TABLE=$TABLE"
echo "LOCAL_IP=$LOCAL_IP"
echo "EGRESS=$EGRESS_JSON"
```

### Логика парсинга в боте

```python
def parse_podkop_status(out: dict) -> str:
    sing = out["SING"] == "1"
    table = out["TABLE"] == "1"
    local_ip = out["LOCAL_IP"]
    egress = json.loads(out["EGRESS"]) if out["EGRESS"] else None

    # Базовый детектор: процесс + nftables-таблица
    if not sing or not table:
        return "🔴 STOPPED"

    # Усиленный детектор: трафик реально идёт через VLESS
    if egress and local_ip and egress.get("ip") != local_ip:
        return "🟢 RUNNING"

    # Процесс есть, но маршруты не работают
    if egress and local_ip and egress.get("ip") == local_ip:
        return "🟡 RUNNING (но трафик идёт мимо VLESS)"

    # Один из IP-запросов упал — показываем что есть
    return "🟢 RUNNING (egress не проверился)"
```

### Дополнительный плюс

Этот же вызов даёт данные для пункта 3 (отображение IP/страны).
Один SSH-раундтрип — и status, и оба IP, и страна. Кешируем результат
в таблице `egress_cache` на 10 минут (см. схему БД ниже).

### Артефакты диагностики

Эталонные снимки состояний (id02, sing-box+podkop версии на апрель 2026):

| Сигнал | RUNNING | STOPPED |
|---|---|---|
| `pgrep -f sing-box` | один процесс | пусто |
| `nft list tables` | `fw4 + PodkopTable` | только `fw4` |
| `nft list sets \| grep podkop` | `podkop_subnets`, `podkop_discord_subnets` | пусто |
| `/var/run/podkop_list_update.pid` | существует (но это не daemon-PID, а лок job-а обновления списков — не использовать) | отсутствует |
| `/etc/init.d/podkop status` | `not running` exit=5 | `not running` exit=5 |
| `ifconfig.co/json country` | egress (DE/NL/US/...) | RU (или страна провайдера) |
| `yandex.ru/internet/api/v0/ip/` | RU IP провайдера | RU IP провайдера (тот же) |

# Часть II. Админская часть — спецификация доработок

Все клиентские доработки (пп. 1–8 выше) **переносятся в админский
интерфейс** без дневных лимитов и ограничений. Дополнительно — список
багов и новых фич чисто админских.

Приоритет реализации: **сначала клиентские пункты 5,6 → 1+3 (общий
SSH-вызов используется и в админке) → клиентский 4 → админский A1
(статусы) → A2 (custom VLESS) → A3 (multi-column) → A4 (max 999) → A5
(версии) → клиентский 7 → клиентский 8 → клиентский 2 → A6 (push-
уведомления) → A7 (аудит-лог)**.

---

## Принцип: всё клиентское доступно админу

Админ в меню любого роутера видит **те же кнопки и фичи, что клиент**,
но:

- **без дневных лимитов** (не 5 логов / не 20 add-del routing rules);
- **без двойных подтверждений** на низкорисковых действиях
  (перезагрузка службы — одно подтверждение, перезагрузка роутера и
  смена WiFi — двойное всё равно остаётся);
- **с расширенной info-плашкой** (см. A5 — версии, модель, sing-box);
- **с правом редактировать клиентские WiFi и routing rules** —
  изменения логируются в `audit_log` с пометкой `actor='admin'`,
  клиент в своей истории WiFi видит запись «изменено админом».

В коде это реализуется через middleware-проверку `is_admin(user_id)`:
если True — лимиты обходятся, дополнительные кнопки рисуются.

## A1. Список «Все роутеры» — статусы и фильтры

### Что было не так

Сейчас бот показывает только `online` роутеры. Если роутер только что
получен курьером и едет к покупателю, или временно выведен на
техработы — его не видно в админском списке вообще, что мешает
управлению парком.

### Новая модель статусов (lifecycle)

Хранятся **в `bot.db`** в таблице `routers`. Поле `lifecycle_status`
с допустимыми значениями:

| Статус | Цвет | Описание |
|---|---|---|
| `online` | 🟢 зелёный | FRP-туннель активен, SSH отвечает |
| `offline` | 🔴 красный | Не отвечает >5 минут (FRP disconnect) |
| `transit` | 🟣 фиолетовый | «Едет к покупателю» — выставляется
  админом вручную при отправке курьеру. Бот не пытается ходить SSH. |
| `maintenance` | 🛠 серый | «Техработы» — только админ может
  включить/выключить вручную. Бот не пытается ходить SSH. |

Поле `lifecycle_status` управляется только через UI бота. В
`config.yaml` хранится только структура (id, label, имя). Это
осознанное решение — конфиг не трогаем при штатных операциях,
снижаем риск рассинхронизации.

### Переходы

```
[transit] --(админ кнопкой "✅ Активировать")--> [online]
[online]  --(FRP disconnect >5 мин, авто)--> [offline]
[offline] --(FRP reconnect, авто)--> [online]
[online]  --(админ кнопкой "🛠 Техработы")--> [maintenance]
[maintenance] --(админ кнопкой "▶ Снять техработы")--> [online]
```

**Авто-перехода `transit → online` НЕТ**. Даже если роутер уже
подключился по FRP, статус остаётся `transit` пока админ не нажмёт
«✅ Активировать» — это сознательный шаг подтверждения «клиент получил
посылку, можно отдавать в работу».

`maintenance` тоже **не снимается автоматически** — только админ
руками. Это защищает от ситуации «починили на 2 минуты, оно само
переключилось в online, клиент пошёл, опять упало».

### UI «Все роутеры» (админ)

- **3 кнопки в ряд** (компактно для 50+ роутеров на мобильнике).
- На каждой кнопке: цветной emoji статуса + `id<NN>` + первые 4
  буквы owner-username, если есть.
- Сверху меню — фильтры: `🟢 online (12)`, `🔴 offline (3)`,
  `🟣 transit (2)`, `🛠 maint (1)`, `📋 все (18)`. Цифра в скобках —
  количество. По умолчанию открыт фильтр `все`.
- Под фильтрами — кнопка `🔍 Поиск по id/owner` (текстовый ввод).

### Кнопки в меню конкретного роутера (админ)

Дополнительно к клиентским кнопкам:

- `🛠 Техработы` (если статус `online` или `offline`)
- `▶ Снять техработы` (если статус `maintenance`)
- `🚚 В пути` (перевести в `transit`, например при возврате от
  клиента)
- `✅ Активировать` (если статус `transit`)
- `🔧 Custom VLESS` (см. A2)
- `📜 Аудит роутера` (см. A7)

### Сообщения клиенту

- Если статус `maintenance` — клиент видит свой роутер с серой
  плашкой и текстом `🛠 Техработы, попробуйте позже` вместо обычного
  меню. Кнопок действий нет, кроме `🔄 Обновить статус` (раз в
  минуту).
- Если статус `transit` — клиент **не видит** этот роутер в своём
  списке. Появится только после `активации` админом.
- Если статус `offline` — клиент видит красную плашку «Роутер не
  отвечает» + кнопку `📨 Сообщить админу» (отправляет push админу).
- **Клиент никогда не видит чужих роутеров** — только те, что
  привязаны к его TG-ID в таблице `router_owners`.

## A2. Custom VLESS вручную (без пула)

### Что было не так

Сейчас VLESS-ключи берутся только из общего пула (`vless_pool.yaml`).
Если нужно подсунуть какому-то роутеру специфический ключ от другого
провайдера VPN или новый тестовый — приходится либо подкладывать в
пул (и тогда он попадёт случайному другому роутеру при ротации),
либо лазить SSH вручную.

### Решение

Кнопка `🔧 Custom VLESS` в админском меню роутера. Сценарий:

1. Админ жмёт кнопку → появляется текстовое поле «Вставьте VLESS
   URL (формат `vless://...`)».
2. Бот валидирует строго (regex + парсинг полей `id`, `host`, `port`,
   `params.security=reality`, `params.pbk`, `params.sni`).
3. **Двойное подтверждение** с показом нового URL в укороченном
   виде (`vless://…@de.example.com:443/...`).
4. После подтверждения:
   - Запись в `routers.custom_vless_url` (полный URL).
   - SSH на роутер: `uci set podkop.@vless[0].url='<URL>'; uci commit
     podkop; /etc/init.d/podkop restart`.
   - Запись в `audit_log` с маской на `id` и `pbk` (последние 4
     символа видны).
5. Роутер **выходит из пула ротации** (поле `in_pool=0`). Если
   планируется ротация — этот роутер скипается.
6. Возврат в пул — отдельная кнопка `🔁 Вернуть в пул`. Тогда
   `custom_vless_url` обнуляется, на роутер кладётся следующий ключ
   из пула.

### Как клиент видит custom

В клиентском info-экране (см. п.4 клиентской части) показывается
страна+домен — без указания, custom это или из пула. Для клиента
прозрачно.

## A3. Multi-column layout — окончательно

3 кнопки в ряд (см. A1). При 50 роутерах = 17 рядов. На мобильнике
влезает примерно 9-10 рядов на экран — листать придётся, но это
нормально и не вызывает фрустрации (как пробовали 5 в ряд — текст
обрезался).

При 999 роутерах админский UI **обязательно** должен использовать
фильтры и поиск (см. A1) — листать 333 ряда нереально.

## A4. Лимит роутеров — 999

Меняем validation в `config.yaml` loader: `max_routers=999`.
Идентификаторы остаются переменной длины: `id1, id2, id07, id99,
id100, id999`. **Миграции id-формата не делаем** — это сломает
подстановки в логах, в FRP-туннелях, в путях к ключам, в шапках
сообщений админу. Единственный косметический минус — несимметричное
выравнивание в списках, с этим живём.

## A5. Версии и железо в info-экране админа

В info-экране роутера для админа дополнительно показываются:

- `model` — `cat /tmp/sysinfo/model` или `ubus call system board |
  jsonfilter -e '@.model'`. Кеш в `routers.hw_model` (постоянный,
  обновляется только при `🔄 Обновить данные`).
- `openwrt_version` — `. /etc/openwrt_release && echo
  $DISTRIB_RELEASE`. Кеш в `routers.openwrt_version`. Обновляется
  раз в сутки автоматически + при ручном refresh.
- `podkop_version` — `opkg list-installed | grep ^podkop` или
  `cat /usr/share/podkop/version` если есть. Кеш в
  `routers.podkop_version`.
- `singbox_version` — `/usr/bin/sing-box version | head -n1`. Кеш
  в `routers.singbox_version`.

Все версии собираются **одним SSH-вызовом** вместе с детектором
статуса (см. п.1) при принудительном refresh — не дёргаем роутер
лишний раз. Раз в сутки фоновый job обновляет версии у всех `online`
роутеров для отчётности.

### Формат показа

```
ℹ id07 — owner: @yarik88
🟢 online · DE Frankfurt · vless: de1.example.com
─────────────────
Модель:    Xiaomi Redmi AC2100
OpenWrt:   23.05.2
podkop:    1.4.7
sing-box:  1.10.5
─────────────────
[🔄 Обновить данные]
```

## A6. Push-уведомления админу

### Что отправляем

Админ получает push в Telegram **только в двух случаях**:

1. **Новый клиент подключился к боту.** Когда клиент впервые жмёт
   `/start` и его TG-ID находит совпадение с `router_owners` —
   приходит уведомление:

   ```
   👤 Новый клиент: @username (TG: 123456789)
   Привязан к: id07
   Статус роутера: 🟢 online
   ```

   Если клиент не находится в `router_owners` — админу приходит
   слабая нотификация (низкий приоритет) с кнопкой «Привязать к…».

2. **Клиент изменил WiFi или routing.** Любая успешная смена SSID,
   пароля, добавление/удаление routing rule:

   ```
   ⚙ id07 (@username): сменил WiFi
   Было: HomeNet → Стало: Family5G
   [📜 История изменений]
   ```

   Кнопка `📜 История изменений` ведёт в админский audit-лог по
   этому роутеру.

### Что НЕ отправляем

- ❌ Падение/восстановление роутера (offline/online события). Иначе
  при 50 роутерах с 4G-связью админу прилетит 200 пингов в день. Если
  понадобится — сделаем отдельной фичей с порогом «упало >30 минут».
- ❌ Отправка лога теста клиентом — это идёт по отдельному каналу
  (см. клиентский п.2), не через push-сценарий.
- ❌ Перезагрузки службы/роутера клиентом — слишком частая операция.

### Доставка

Все push-уведомления идут одному `admin_chat_id` из `bot.yaml`. Если
админов несколько — поддержать `admin_chat_ids: [...]` (список).

## A7. Audit-лог — 90 дней

### Таблица `audit_log`

```sql
CREATE TABLE audit_log (
  id          INTEGER PRIMARY KEY AUTOINCREMENT,
  ts          INTEGER NOT NULL,             -- unix epoch
  actor_type  TEXT NOT NULL CHECK(actor_type IN ('client','admin','system')),
  actor_id    INTEGER,                      -- TG user_id (NULL для system)
  router_id   TEXT,                         -- 'id07' или NULL
  action      TEXT NOT NULL,                -- 'wifi_change', 'restart_router',
                                            -- 'restart_service', 'routing_add',
                                            -- 'routing_del', 'vless_change',
                                            -- 'maintenance_on', 'lifecycle_change'
  payload     TEXT,                         -- JSON с before/after
  result      TEXT NOT NULL CHECK(result IN ('ok','fail')),
  error       TEXT
);
CREATE INDEX idx_audit_router_ts ON audit_log(router_id, ts DESC);
CREATE INDEX idx_audit_ts        ON audit_log(ts);
```

### Retention

Cron-задача в боте раз в сутки в 04:00:

```sql
DELETE FROM audit_log WHERE ts < strftime('%s','now') - 90*86400;
VACUUM;
```

90 дней — компромисс между разбором инцидентов («что было два месяца
назад») и размером базы (~50 КБ × 50 роутеров × 90 дней ≈ 225 МБ
максимум, на практике в 5–10 раз меньше).

### Просмотр

В админском меню роутера: `📜 Аудит роутера` → последние 30 записей
с пагинацией. Формат:

```
2026-04-23 14:32 · 🛠 admin · maintenance_on
2026-04-23 12:15 · 👤 @yarik88 · wifi_change
   SSID: HomeNet → Family5G
2026-04-22 23:01 · 👤 @yarik88 · routing_add
   Domain: youtube.com → podkop
```

Кнопка `📥 Экспорт CSV` — выгружает аудит за последние 90 дней
файлом.

## A8. Глобальный тест — финальный формат

**Решение пользователя (override клиентского п.2):** оставляем
текущий формат — **полный лог отправляется файлом**, не разбивается
на сообщения.

### Параметры запуска

- Тестируются **только `online` роутеры** (не purple, не red, не
  maintenance). Это решение для масштаба 50–999.
- Параллелизм: до 5 роутеров одновременно (SSH-семафор).
- Таймаут на роутер: 30 секунд.

### Формат отчёта

Текстовый файл `global_test_<YYYY-MM-DD_HH-MM>.log` с разделами на
каждый роутер:

```
======================================
id07  · owner: @yarik88  · DE Frankfurt
======================================
[OK] sing-box процесс жив
[OK] PodkopTable в nftables
[OK] local_ip: 176.15.166.202 (RU MTS)
[OK] egress_ip: 89.149.x.x (DE Frankfurt)
[OK] DNS resolve youtube.com → 142.250.x.x
[OK] curl https://youtube.com → 200 OK (1.2s)
RESULT: ✅ PASS
```

В конце файла — сводка:

```
======== SUMMARY ========
Total tested: 47
✅ PASS: 44
⚠ WARN: 2 (id12: egress_ip == local_ip)
❌ FAIL: 1 (id33: sing-box не отвечает)
```

В чат админу приходит **короткое сообщение** + файл:

```
🧪 Глобальный тест завершён
Протестировано: 47/47 онлайн-роутеров
✅ 44  ⚠ 2  ❌ 1
[📎 global_test_2026-04-23_14-30.log]
```

## A9. Модель привязки: 1 аккаунт = N роутеров

### Структура

Таблица `router_owners`:

```sql
CREATE TABLE router_owners (
  router_id   TEXT NOT NULL,
  tg_user_id  INTEGER NOT NULL,
  is_primary  INTEGER NOT NULL DEFAULT 0,  -- 1 = «основной» владелец
  added_at    INTEGER NOT NULL,
  added_by    INTEGER NOT NULL,            -- TG-ID админа
  PRIMARY KEY (router_id, tg_user_id)
);
CREATE INDEX idx_owners_user ON router_owners(tg_user_id);
```

Один роутер может иметь несколько TG-владельцев (семья), один TG-юзер
может иметь несколько роутеров (квартира + дача).

### Клиентский UX

- Если у клиента **1 роутер** — после `/start` сразу открывается
  меню этого роутера, как сейчас. Без шага выбора.
- Если у клиента **N >= 2 роутеров** — сначала показывается список
  его роутеров с цветовыми статусами, кнопка → меню роутера.

### Админский UX

- В меню роутера показываются все его владельцы:
  ```
  Владельцы: @yarik88 (основной), @wife88
  [➕ Добавить] [➖ Убрать]
  ```
- Добавление: админ вводит TG-ID или username (если бот видел
  юзера). Один из владельцев должен быть `is_primary=1` — на него
  идут уведомления (например, о смене VLESS).
