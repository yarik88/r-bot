# Безопасность r-bot

Этот документ — не параноидальная теория, а практический чеклист. Репозиторий
публичный, а на сервере лежат секреты с очень высокой ценой компрометации:
**SSH-пароль root'а от 52 чужих роутеров** и **токен Telegram-бота**, через
который можно отдавать команды этим роутерам. Утекло — и парк чужой.

## Что считается секретом

| Значение | Где живёт | Что будет, если утечёт |
|---|---|---|
| `ssh.password` в `/etc/r/config.yaml` | только на сервере | чужой получает root на всех 52 роутерах — полная компрометация парка |
| `token` в `/etc/r/bot.yaml` | только на сервере | чужой выдаёт себя за твоего бота, но без `admins` нашедших свои id в whitelist — урона не нанесёт (middleware блокирует). Всё равно меняй. |
| Telegram user id в `admins:` | условный секрет | сам по себе не опасен, но в связке с утёкшим токеном даёт злоумышленнику быстрый путь к админ-правам. Не публикуй. |
| `vless://...`-ссылки в пуле (sqlite) | `/var/lib/r-bot/bot.db` | чужой подключается к твоим VLESS-серверам, ест трафик. |
| Публичные IP серверов в `config.yaml`/логах | сервер | облегчают атаку (сканы, брутфорс SSH). Не публикуй. |

## Защита на сервере

Права на конфиги — **фиксированно 600 root:root**:

```bash
sudo chmod 600 /etc/r/bot.yaml /etc/r/config.yaml
sudo chown root:root /etc/r/bot.yaml /etc/r/config.yaml
sudo chmod 700 /etc/r
```

База данных — тоже:

```bash
sudo chmod 600 /var/lib/r-bot/bot.db
sudo chmod 700 /var/lib/r-bot
```

systemd unit запускает бота под `root`, но с набором защит
(`ProtectSystem=full`, `NoNewPrivileges`, `MemoryDenyWriteExecute`,
`MemoryMax=256M`, `CPUQuota=50%`). Это снижает поверхность атаки даже в
случае RCE через Telegram.

`frps` слушает туннели **только на loopback** (`127.0.0.1:11XXX`). Снаружи
порты не открыты. Если `ss -tlnp` показывает `0.0.0.0:11XXX` — это ошибка
конфига frps, чини до запуска бота.

## Защита репозитория

Репо публичный. Правила простые:

1. **Ни одной настоящей секретной строки в коде и `.example`.** Только
   плейсхолдеры типа `PASTE_YOUR_BOT_TOKEN_HERE` и комментарий-объяснение.
2. **`.gitignore` уже покрывает** `config/bot.yaml`, `config/config.yaml`,
   `*.db`, `*.log`, `r-bot_*.tar.gz`. Не правь этот список, не вникнув.
3. **Перед каждым `git push`** — `bash scripts/preflight.sh`. Он сканирует
   tracked-файлы на:
   - телеграм-токены вида `\d{6,}:[A-Za-z0-9_-]{30,}`
   - непустые `password:` со значением ≥ 4 символов
   - живые `vless://uuid@host`
   - публичные IP-подсети, которые ты используешь (в скрипте они перечислены —
     допиши свои при развёртывании)
   - случайно застейженные `*.db`, `*.log`, `*.tar.gz`
4. Если бот, работая через кнопку `🔄 Обновить из Git`, делает `git pull` —
   он **читает** из репо, никогда не **пишет** в него. Push делается только
   с твоей машины руками, после preflight.

## Что делать, если секрет утёк

### Телеграм-токен

1. `@BotFather` → `/revoke` на этого бота → выдаёт новый токен.
2. `sudo nano /etc/r/bot.yaml`, замени `token: "..."`.
3. `sudo systemctl restart r-bot`.
4. Проверь `journalctl -u r-bot -n 30` — должно быть `starting polling...`.

### SSH-пароль роутеров

Тяжелее, потому что пароль один на весь парк:

1. Составь план смены пароля на всех 52 роутерах (можно массово через
   бота: открой global_check → подтверди YES, но шаблон команды поменяй на
   `passwd root` — это придётся сделать руками в коде разово, или
   через твой TUI `r`).
2. Поменяй пароль на роутерах.
3. Обнови `/etc/r/config.yaml` → `ssh.password:`.
4. `sudo systemctl restart r-bot`.

### Бэкап попал в публичный репо

1. Удали файл из истории git: `git filter-repo --path backups/... --invert-paths` (или `bfg`).
2. `git push --force-with-lease` (это как раз тот случай, когда force оправдан).
3. Сразу поменяй **всё**, что было в этом бэкапе: токен бота, пароль роутеров,
   VLESS-ссылки (на сервере-провайдере).

## Audit

Все деструктивные действия (reboot, podkop restart, смена VLESS, массовые
операции, смена ярлыков, операции с клиентами) пишутся в табличку `audit`
внутри `bot.db` с полями `ts / user_id / router_id / action / args / success`.

Посмотреть глазами:

```bash
sudo sqlite3 /var/lib/r-bot/bot.db \
  "SELECT datetime(ts,'unixepoch','localtime'), user_id, router_id, action, success \
   FROM audit ORDER BY ts DESC LIMIT 50;"
```

Бот сам чистит записи старше 90 дней (см. `job_cleanup_runs`), если хочешь
держать дольше — поправь в `r_bot.py`.

## Чего бот НЕ делает (по дизайну)

- Не выполняет произвольные shell-команды, присланные клиентом. Только
  фиксированный список операций через кнопки.
- Не принимает vless-ссылки от клиентов. Пул наполняет только admin.
- Не хранит пароли/токены клиентов в логах и в audit'е (только тип действия).
- Не отдаёт `bot.db` и `config.yaml` через кнопку «Бэкап», если запрос не
  от admin'а. Клиентам кнопки бэкапа вообще не видно.
- Не падает в open-connection на роутер дольше таймаута (`ssh_timeout_sec`),
  чтобы нельзя было DoS'ить бота массовыми ping'ами.

## Что в планах (но пока НЕ реализовано)

- Шифрование `ssh.password` at-rest (сейчас — просто 600-права на файл).
- 2FA на входе клиента (сейчас — только whitelist по user_id).
- Webhook вместо long polling (пока не нужен; long polling работает без
  публичного домена, что само по себе плюс).
- Role `viewer` (только чтение, без действий) — если появится запрос.
